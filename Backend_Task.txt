2.4  BullMQ Jobs Setup
	•	Install: bullmq
	•	Create src/jobs/provision.job.ts — BullMQ Worker listening on queue 'provision'
	•	  Job payload: { sessionId, labId, userId, dockerImage, initScript, exposePort }
	•	  Step 1: call container.service.createContainer + startContainer
	•	  Step 2: wait for container healthcheck (poll inspectContainer until running, max 60s)
	•	  Step 3: call auth.service.issueEmbedToken(sessionId, userId) to get signed terminal token
	•	  Step 4: build terminal_url: https://{sessionId}.labs.yourdomain.com/terminal?token={embedToken}
	•	  Step 5: update DB session: status='ready', terminal_url, ready_at=NOW()
	•	  Step 6: publish Redis event lab:{sessionId}:events → { type: 'session_ready', terminal_url }
	•	  On any error: update DB session status='error', error_message. Remove container if it started.
	•	  Set job timeout: 60s (failsafe — no hanging provisioning state)
	•	Create src/jobs/cleanup.job.ts — BullMQ repeating job every CLEANUP_INTERVAL_MS
	•	  Query DB: sessions WHERE expires_at < NOW() AND status NOT IN ('destroyed','expired')
	•	  For each: call orchestrator.destroyLab(sessionId). Update status=expired.
	•	Create src/jobs/monitor.job.ts — BullMQ repeating job every MONITOR_INTERVAL_MS
	•	  For each active session: call container.service.getContainerStats
	•	  Insert row into resource_snapshots table
	•	  If cpu_percent > 90: publish resource_warning WebSocket event

2.5  Lab Orchestrator — src/services/orchestrator.service.ts
	•	Implement provisionLab(userId, labId, options):
	•	  1. Count active sessions for user. Reject with QUOTA_EXCEEDED if >= MAX_ACTIVE_SESSIONS_PER_USER
	•	  2. Check idempotency: if session status IN ('provisioning','ready') for user+lab → return existing
	•	  3. Count ALL active containers. Reject with LAB_CAPACITY_FULL if >= MAX_TOTAL_ACTIVE_CONTAINERS
	•	  4. INSERT session row: status='provisioning', expires_at=NOW()+DEFAULT_SESSION_TTL
	•	  5. SET Redis: sess:{sessionId} = '1' EX DEFAULT_SESSION_TTL
	•	  6. Enqueue provision.job to BullMQ with session payload
	•	  7. Return session record immediately (async — do not await container start)
	•	Implement destroyLab(sessionId):
	•	  1. Load session from DB, verify not already destroyed
	•	  2. Call container.service.stopContainer + removeContainer (handle not-found gracefully)
	•	  3. DEL Redis key sess:{sessionId}
	•	  4. UPDATE DB: status='destroyed', ended_at=NOW()
	•	  5. Publish session_expired WebSocket event
	•	Implement getSessionStatus(sessionId, userId): load from DB, verify ownership, return status + terminal_url

2.6  Session Service — src/services/session.service.ts
	•	Implement heartbeat(sessionId, userId): verify session in DB, SET Redis sess:{sessionId} EX DEFAULT_SESSION_TTL
	•	Implement extendSession(sessionId, extendBySeconds): add time, cap at MAX_SESSION_TTL from created_at
	•	  Update Redis TTL + DB expires_at in the same operation
	•	  Reject if requested extension would exceed MAX_SESSION_TTL total

PHASE 4 — Auth System  (Start in Parallel with Person 1's Phase 1)
ℹ  Auth has zero dependency on Docker or containers. Start this immediately once the repo + DB schema exist.

4.1  Auth Service — src/services/auth.service.ts
	•	Install: jose, bcrypt, @types/bcrypt, zod
	•	Implement hashPassword(plain): bcrypt.hash with 12 rounds
	•	Implement verifyPassword(plain, hash): bcrypt.compare — returns boolean
	•	Implement issueAccessToken(userId, role): sign JWT with HS256, exp=JWT_ACCESS_EXPIRES (15min)
	•	  Payload: { sub: userId, role, type: 'access' }
	•	Implement issueRefreshToken(userId): sign JWT, store token family ID in Redis SET with 7d TTL
	•	  Payload: { sub: userId, family: randomUUID(), type: 'refresh' }
	•	  Redis key: refresh:{userId}:{family} — store with EX JWT_REFRESH_EXPIRES
	•	Implement issueEmbedToken(sessionId, userId): scoped JWT, exp=LAB_TOKEN_EXPIRES (10min)
	•	  Payload: { sub: userId, session_id: sessionId, scope: 'embed', type: 'embed' }
	•	Implement verifyToken(token): decode + verify, throw TOKEN_EXPIRED or UNAUTHORIZED AppError
	•	Implement refreshTokens(refreshToken): verify, check family in Redis, invalidate old, issue new pair
	•	  If family not in Redis (already rotated): revoke ALL families for user — possible replay attack
	•	Implement revokeAllUserTokens(userId): delete all refresh:{userId}:* keys from Redis
	•	Embed tokens REJECTED on all API endpoints — enforced in authenticate middleware by checking scope field
	•	Never store access tokens in Redis — they are stateless, verified by signature only

4.2  Auth Routes — src/routes/auth.routes.ts
	•	Register with prefix /api/v1/auth
	•	POST /login: Zod body { email: string, password: string }
	•	  Load user by email from DB
	•	  Call verifyPassword — return same 401 INVALID_CREDENTIALS whether email OR password wrong
	•	  On success: issue access + refresh tokens, update user.last_login
	•	POST /refresh: Zod body { refresh_token: string } → call refreshTokens
	•	POST /logout: revoke all tokens for request.user.sub → 200
	•	POST /register: Zod body { email, password, name } → hashPassword, INSERT user
	•	Apply rate limit to /login: 5 requests per IP per 60s
	•	Write TS-01 tests (T01-T12) in tests/ts-01-auth.test.ts
	•	  Use fastify.inject() — no real HTTP calls needed
	•	  Mock DB for unit tests. Use real postgres/redis in integration tests
	•	  Test T11 cross-session embed token rejection explicitly

4.3  Middleware — src/middleware/
	•	Create authenticate.ts — Fastify preHandler hook:
	•	  Extract 'Bearer {token}' from Authorization header
	•	  Call auth.service.verifyToken
	•	  Reject if token.scope === 'embed' (embed tokens forbidden on API routes)
	•	  Set request.user = decoded payload
	•	  On any failure: throw AppError UNAUTHORIZED or TOKEN_EXPIRED
	•	Create ownerGuard.ts — Fastify preHandler hook:
	•	  Read request.params.sessionId
	•	  Query DB: SELECT user_id FROM sessions WHERE id = sessionId
	•	  If session.user_id !== request.user.sub → throw AppError FORBIDDEN
	•	Create errorHandler.ts — Fastify setErrorHandler:
	•	  Map AppError.code to HTTP status codes (see error table in frontend doc)
	•	  Never expose stack traces when NODE_ENV === 'production'
	•	  Return { error: { code, message, http_status } } shape — matches frontend error handling
